"Stator optimizer".

Fully optimizes one "strip" at a time, in random order, until it cannot
optimize whatever it has any further.

Expects pattern as periods and asterisks on standard in, see `examples` for
some example input.  Outputs ... a lot of crap.  Changes to the pattern are
output with `x` for deleted cells, `o` for added cells, and `r`/`R` for rotor
cells.  Similarly the final optimized pattern is output with `r`/`R` for rotor
cells.

Arguments are:

(1) number of blanks to pad the bounding box with.  2-4 is probably good enough
to find any real optimizations.  0 if you want to refuse increasing the
bounding box.

(2) Width of search strips.  Depending on your available CPU and RAM, and
willingness to wait, your best choice here will vary.  5 is very fast (a few
seconds even for large patterns), but will definitely miss some optimizations.
9 will "fix" the worst local optimum I currently know (`examples/2xx-bad.in`)
but takes a bit (~30s to fix just that small example).
